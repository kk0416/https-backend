# 接口联调说明

## 1. 文档目的

1. 说明当前已经可以联调的后端接口。
2. 说明前端在联调时应该先调哪些接口，再调哪些接口。
3. 说明当前返回结构、错误结构和枚举格式。
4. 目标读者是 `Qt/C++` 前端开发者。

## 2. 当前联调地址

### 2.1 云上地址

当前已经验证可访问：

`https://sep-gaussian-backend.cloud-data-dev.seer-group.com`

### 2.2 当前接口前缀

业务接口统一前缀：

`/api/v1`

### 2.3 当前健康检查地址

1. `GET /health`
2. `GET /api/v1/health`

完整地址分别是：

1. `https://sep-gaussian-backend.cloud-data-dev.seer-group.com/health`
2. `https://sep-gaussian-backend.cloud-data-dev.seer-group.com/api/v1/health`

## 3. 当前联调前提

### 3.1 当前没有登录鉴权

第一版当前没有：

1. 登录接口
2. Token
3. 用户权限校验

所以当前联调时，不需要额外带 `Authorization` 请求头。

### 3.2 当前证书说明

当前环境的 HTTPS 已经可访问，但证书是测试阶段手工生成的。

这意味着：

1. `curl` 访问时可以临时使用 `-k`
2. Qt 联调时，如果证书未被系统信任，可能会收到 SSL 错误
3. 当前阶段可以临时忽略 SSL 错误做联调
4. 正式版本建议替换为受信任证书，不要长期忽略 SSL 错误

## 4. 统一返回结构

### 4.1 成功返回

所有成功接口都统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 4.2 错误返回

当前错误接口统一返回：

```json
{
  "code": 400,
  "message": "invalid data asset id",
  "data": {
    "path": "/api/v1/data-assets/abc/generate-point-cloud",
    "timestamp": "2026-03-27T10:00:00.000Z"
  }
}
```

### 4.3 前端处理建议

建议前端统一按下面规则处理：

1. `code == 0` 视为成功
2. `code != 0` 视为失败
3. 直接展示 `message`
4. 不要把 HTTP `200` 当成唯一成功判断条件

## 5. 枚举格式约定

### 5.1 API 返回值格式

数据库里虽然是大写枚举，但 API 返回给前端时，统一转成小写形式。

例如：

1. `RAW -> raw`
2. `POINT_CLOUD -> point_cloud`
3. `MAP_2D -> map_2d`
4. `READY -> ready`
5. `RUNNING -> running`

### 5.2 前端查询参数格式

前端传筛选参数时，也建议直接传小写：

1. `dataType=raw`
2. `dataType=point_cloud`
3. `status=ready`
4. `status=running`

## 6. 当前已开放接口

### 6.1 健康检查

1. `GET /health`
2. `GET /api/v1/health`

返回示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "service": "gaussian-backend-demo",
    "status": "running"
  }
}
```

### 6.2 数据总览

接口：

`GET /api/v1/dashboard/data-summary`

兼容别名：

`GET /api/v1/data-summary`

可选参数：

1. `siteId`
2. `sceneId`

示例：

`GET /api/v1/dashboard/data-summary?siteId=1&sceneId=1`

或：

`GET /api/v1/data-summary?siteId=1&sceneId=1`

返回结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "rawCount": 1,
    "pointCloudCount": 1,
    "gaussianCount": 1,
    "map2dCount": 1,
```

### 6.4 树状图

接口：

`GET /api/v1/data-assets/tree`

可选参数：

1. `siteId`
2. `sceneId`

返回结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": [
    {
      "id": 1,
      "dataName": "仓库原始数据-01",
      "dataType": "raw",
      "status": "ready",
      "progress": 100,
      "sourceDataId": null,
      "children": [
        {
          "id": 2,
          "dataName": "仓库点云数据-01",
          "dataType": "point_cloud",
          "status": "ready",
          "progress": 100,
          "sourceDataId": 1,
          "children": []

    }
```

说明：

1. 根节点就是 `sourceDataId = null` 的数据
2. 当前树完全由 `sourceDataId` 关系生成
3. 当前没有单独关系表

### 6.5 关系图

接口：

`GET /api/v1/data-assets/graph`

可选参数：

1. `siteId`
2. `sceneId`

返回结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "nodes": [
      {
        "id": 1,
        "label": "仓库原始数据-01",
        "dataType": "raw",
        "status": "ready",
        "progress": 100,
        "siteId": 1,
        "sceneId": 1
      }
    ],
    "edges": [
      {
        "source": 1,
        "target": 2
      }
    ]
  }
}
```

说明：

1. `nodes` 直接给图控件使用
2. `edges` 表示从源数据指向目标数据
3. 这类结构对网页图形库和 Qt 图形场景都比较容易消费

### 6.7 生成点云任务

接口：

`POST /api/v1/data-assets/:id/generate-point-cloud`

说明：

1. 当前不需要请求体
2. `:id` 必须是一个已有的 `raw` 类型数据 ID

成功返回示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "taskId": 5,
    "targetDataId": 6,
    "status": "queued"
  }
}
```

常见失败情况：

1. `id` 不是数字：`invalid data asset id`
2. 数据不存在：`data asset not found`
3. 数据不是 `raw`：`only raw data can generate point cloud`
4. 数据已删除：`deleted data cannot create tasks`

## 7. 推荐联调顺序

### 7.1 第一步

先确认服务在线：

1. `GET /health`
2. `GET /api/v1/health`

### 7.2 第二步

再调不带副作用的只读接口：

1. `GET /api/v1/dashboard/data-summary`
2. `GET /api/v1/data-assets?page=1&pageSize=20`
3. `GET /api/v1/data-assets/tree`
4. `GET /api/v1/data-assets/graph`
5. `GET /api/v1/tasks?page=1&pageSize=20`

### 7.3 第三步

从数据列表里找一个 `dataType = raw` 的数据 ID，再调：

`POST /api/v1/data-assets/:id/generate-point-cloud`

### 7.4 第四步

生成任务之后，再刷新：

1. `GET /api/v1/data-assets`
2. `GET /api/v1/tasks`
3. `GET /api/v1/data-assets/tree`
4. `GET /api/v1/data-assets/graph`

## 8. 适合 Qt 前端的最小联调方式

### 8.1 推荐类

建议先用：

`QNetworkAccessManager`

### 8.2 一个最小 GET 示例

```cpp
QNetworkAccessManager* manager = new QNetworkAccessManager(this);

QUrl url("https://sep-gaussian-backend.cloud-data-dev.seer-group.com/api/v1/health");
QNetworkRequest request(url);
request.setHeader(QNetworkRequest::ContentTypeHeader, "application/json");

QNetworkReply* reply = manager->get(request);

connect(reply, &QNetworkReply::finished, this, [reply]() {
    const QByteArray body = reply->readAll();
    qDebug() << body;
    reply->deleteLater();
});
```

### 8.3 一个最小 POST 示例

```cpp
QNetworkAccessManager* manager = new QNetworkAccessManager(this);

QUrl url("https://sep-gaussian-backend.cloud-data-dev.seer-group.com/api/v1/data-assets/1/generate-point-cloud");
QNetworkRequest request(url);
request.setHeader(QNetworkRequest::ContentTypeHeader, "application/json");

QNetworkReply* reply = manager->post(request, QByteArray());

connect(reply, &QNetworkReply::finished, this, [reply]() {
    const QByteArray body = reply->readAll();
    qDebug() << body;
    reply->deleteLater();
});
```

说明：

1. 上面 `1` 只是示例 ID
2. 实际联调时，应该先从 `GET /api/v1/data-assets` 找到真实 `raw` 数据 ID

### 8.4 当前 HTTPS 证书如果不被 Qt 信任

如果当前测试环境证书未被系统信任，Qt 里可能收到 SSL 错误。

测试阶段可临时这样处理：

```cpp
connect(reply, &QNetworkReply::sslErrors, this, [reply](const QList<QSslError>& errors) {
    Q_UNUSED(errors);
    reply->ignoreSslErrors();
});
```

注意：

1. 这只适合当前测试联调
2. 正式环境不建议长期忽略 SSL 错误

## 9. 当前第一版接口边界

### 9.1 已有能力

1. 总览统计
2. 数据列表
3. 树状图数据
4. 关系图数据
5. 任务列表
6. 生成点云任务

### 9.2 暂时没有的接口

当前第一版还没有：

1. 用户登录
2. 用户管理
3. 站点列表接口
4. 场景列表接口
5. 操作日志列表接口
6. 真正异步任务执行器

### 9.3 对前端的实际影响

1. 站点和场景下拉框，如果现在就要做，可以先从数据列表或任务列表里提取 `siteId/siteName`、`sceneId/sceneName`
2. `generate-point-cloud` 当前只会创建任务记录和目标数据记录，不会真的去跑点云算法
3. 所以它更适合现在这一阶段的页面联调，不适合当作真实算法结果接口

## 10. 推荐你现在怎么联调

### 10.1 如果你是网页前端

建议顺序：

1. 先接 `health`
2. 再接 `data-summary`
3. 再接 `data-assets`
4. 再接 `tree`
5. 再接 `graph`
6. 最后接 `tasks`

### 10.2 如果你是 Qt 前端

建议顺序：

1. 先做一个统一的 `HttpClient`
2. 统一解析 `code/message/data`
3. 先用只读接口打通页面
4. 最后再接 `POST generate-point-cloud`
