# 代码入口与调用流程

## 1. 文档目的

1. 说明这个后端项目的代码入口在哪里。
2. 说明一个 HTTP 请求是如何在项目中流转的。
3. 说明主要模块之间的关系。
4. 目标读者是第一次接触 `TypeScript/NestJS` 的 `Qt/C++` 开发者。

## 2. 先记住最核心的一句话

这个项目的基本调用链路是：

1. `main.ts` 负责启动应用
2. `AppModule` 负责装配模块
3. `Controller` 负责接 HTTP 请求
4. `Service` 负责业务逻辑
5. `PrismaService` 负责数据库访问
6. `SQLite` 负责落数据

## 3. 代码入口在哪里

### 3.1 真正入口文件

项目真正入口是：

`src/main.ts`

### 3.2 `main.ts` 做了什么

它主要做这几件事：

1. 用 `NestFactory.create()` 创建 Nest 应用
2. 指定底层 HTTP 服务器为 `Fastify`
3. 设置全局前缀 `/api/v1`
4. 注册全局异常过滤器
5. 监听 `3000` 端口

### 3.3 怎么理解

如果类比 Qt/C++：

1. `main.ts` 很像后端程序的 `main()`
2. `bootstrap()` 很像应用启动初始化流程
3. `app.listen(...)` 很像让服务正式开始对外提供能力

## 4. 根模块在哪里

### 4.1 根模块文件

根模块是：

`src/app.module.ts`

### 4.2 根模块的作用

它负责：

1. 注册所有子模块
2. 告诉 Nest 哪些 controller 和 service 要参与装配
3. 建立整个项目的依赖注入关系

### 4.3 当前已接入的主要模块

1. `PrismaModule`
2. `DashboardModule`
3. `DataAssetModule`
4. `ProcessTaskModule`
5. `OperationLogModule`
6. `AuthModule`

## 5. Prisma 和数据库是怎么接进来的

### 5.1 `src/prisma/prisma-client-options.ts`

这个文件负责创建 Prisma 的运行时配置。

它的作用是：

1. 从 `.env` 读取 `DATABASE_URL`
2. 创建 SQLite adapter
3. 返回给 `PrismaClient` 使用

### 5.2 `src/prisma/prisma.service.ts`

这个文件定义了 `PrismaService`。

你可以把它理解成：

1. 项目里统一的数据库访问对象
2. 所有业务 service 都通过它来查库和写库

### 5.3 `src/prisma/prisma.module.ts`

这个模块把 `PrismaService` 暴露到整个应用中。

因为用了 `@Global()`，所以其他模块中可以直接注入 `PrismaService`。

## 6. 一次请求是怎么流动的

### 6.1 以数据列表接口为例

接口：

`GET /api/v1/data-assets?page=1&pageSize=20`

调用顺序如下：

1. 浏览器或前端发起 HTTP 请求
2. Fastify 收到请求
3. Nest 根据路由把请求分发到 `DataAssetController`
4. `DataAssetController.getList()` 收到参数
5. `DataAssetController` 调用 `DataAssetService.getList()`
6. `DataAssetService` 组装 Prisma 查询条件
7. `DataAssetService` 调用 `PrismaService`
8. `PrismaService` 通过 Prisma Client 查询 SQLite
9. 查询结果返回给 `DataAssetService`
10. `DataAssetService` 把数据库对象转换成前端需要的结构
11. `DataAssetController` 调用 `ok(data)` 包装统一响应
12. 响应返回给前端

### 6.2 用一句话理解

可以把它理解成：

1. `Controller` 是接口入口
2. `Service` 是业务处理中心
3. `PrismaService` 是数据库访问层

## 7. 为什么要分 Controller 和 Service

### 7.1 Controller 负责什么

1. 定义路由
2. 接收参数
3. 调用 service
4. 返回响应

### 7.2 Service 负责什么

1. 业务校验
2. 数据库查询
3. 事务处理
4. 数据结构组装

### 7.3 这样分层的好处

1. controller 更薄，更像接口适配层
2. service 更聚焦业务逻辑
3. 后续更容易维护和扩展

## 9. 生成点云接口的完整业务流

### 9.1 接口入口

接口是：

`POST /api/v1/data-assets/:id/generate-point-cloud`

入口 controller 文件：

`src/modules/data-asset/data-asset.controller.ts`

对应方法：

`generatePointCloud()`

### 9.2 service 里实际做的事情

真正业务在：

`src/modules/data-asset/data-asset.service.ts`

执行顺序如下：

1. 把路径参数 `id` 转成整数
2. 查询源数据是否存在
3. 校验源数据类型是否为 `RAW`
4. 校验源数据是否未删除
5. 开启数据库事务
6. 创建一条 `processTask`
7. 创建一条新的 `POINT_CLOUD` 类型 `dataAsset`
8. 回写任务的 `targetDataId`
9. 回写源数据的 `currentTaskId`
10. 写入一条 `operationLog`
11. 返回任务 ID 和目标数据 ID

### 9.3 为什么这里必须用事务

因为这不是一条单独 SQL，而是一组必须保持一致的操作。

如果中间失败，就可能出现：

1. 任务创建了，但目标数据没创建成功
2. 目标数据创建了，但日志没写成功
3. 源数据没有正确绑定当前任务

所以这里必须放进事务里，保证要么全部成功，要么全部失败。

## 10. 树状图和关系图是怎么构造出来的

### 10.1 树状图

接口：

`GET /api/v1/data-assets/tree`

实现方法：

`DataAssetService.getTree()`

思路：

1. 先查出所有数据资产
2. 基于 `sourceDataId` 判断父子关系
3. 用 `Map<id, node>` 先建立节点索引
4. 再把子节点挂到父节点 `children` 下
5. 没有父节点的节点就是根节点

### 10.2 关系图

接口：

`GET /api/v1/data-assets/graph`

实现方法：

`DataAssetService.getGraph()`

返回结构：

1. `nodes`
2. `edges`

含义：

1. `nodes` 对应 `dataAsset`
2. `edges` 对应 `sourceDataId -> id`

也就是说，当前第一版没有单独关系表，而是直接基于单来源链路生成关系图。

## 11. 全局异常在哪里统一处理

文件：

`src/common/filters/all-exceptions.filter.ts`

作用：

1. 捕获 controller/service 抛出的异常
2. 统一转换成固定 JSON 结构
3. 避免前端拿到不一致的错误格式

例如：

1. `BadRequestException`
2. `NotFoundException`
3. 普通 `Error`

最终都会在这里统一整理。

## 12. 读代码的推荐顺序

如果你是 Qt/C++ 开发者，建议按下面顺序读：

1. `src/main.ts`
2. `src/app.module.ts`
3. `src/prisma/prisma.module.ts`
4. `src/prisma/prisma.service.ts`
5. `src/modules/data-asset/data-asset.controller.ts`
6. `src/modules/data-asset/data-asset.service.ts`
7. `src/modules/process-task/process-task.controller.ts`
8. `src/modules/process-task/process-task.service.ts`
9. `src/modules/dashboard/dashboard.controller.ts`
10. `src/modules/dashboard/dashboard.service.ts`

这样你会先建立“请求是怎么走的”感觉，再看具体业务细节。

你可以把这个项目理解成：
