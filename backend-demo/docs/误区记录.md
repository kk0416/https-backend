# 误区记录

## 1. 文档目的

1. 记录项目推进过程中出现过的关键误区。
2. 记录每个误区对应的正确理解。
3. 后续如果再出现新的误区，继续追加到这份文档中。

## 2. 2026-03-30 误区 01

### 2.1 当时的误区

把当前场景理解成：

1. 和板子有关
2. 多个客户端对应多份数据
3. 需要按客户端或按板子去拆分数据库

### 2.2 正确理解

当前真实场景应理解为：

1. 与板子无关
2. 存储的是各个现场的数据
3. 多个客户端只是访问同一个后端
4. 后端统一查询同一个中心数据库
5. 客户端看到的数据由查询条件决定，不是由客户端本身决定

### 2.3 正确的数据模型理解

当前更合理的理解是：

1. 一套中心后端
2. 一套中心数据库
3. 数据按 `site` 和 `scene` 归类
4. 多个客户端共享访问同一套数据
5. 通过 `siteId`、`sceneId` 过滤查看范围

### 2.4 不应该怎么做

当前不应该：

1. 按客户端拆库
2. 按客户端重复存一份数据
3. 把客户端当成独立数据源

### 2.5 当前结论

当前项目的数据理解应固定为：

`site -> scene -> dataAsset -> processTask / operationLog`

也就是：

1. `site` 表示现场
2. `scene` 表示现场下的场景
3. `dataAsset` 表示场景里的数据资产
4. `processTask` 表示围绕这些数据的处理任务
5. `operationLog` 表示相关操作记录

### 2.6 对后续前端联调的影响

前端后续请求应优先理解为：

1. 客户端只是查看入口
2. 查询重点是当前 `siteId`
3. 查询重点是当前 `sceneId`
4. 如果两个客户端请求条件一样，拿到的数据一样是正常现象
