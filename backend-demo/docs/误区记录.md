## 2. 2026-03-30 误区 01

### 2.1 当时的误区

把当前场景理解成：

1. 和板子有关
2. 多个客户端对应多份数据
3. 需要按客户端或按板子去拆分数据库

### 2.2 正确理解

当前真实场景应理解为：

1. 与板子无关
2. 存储的是各个现场的数据
3. 多个客户端只是访问同一个后端
4. 后端统一查询同一个中心数据库
5. 客户端看到的数据由查询条件决定，不是由客户端本身决定

### 2.3 正确的数据模型理解

当前更合理的理解是：

1. 一套中心后端
2. 一套中心数据库
3. 数据按 `site` 和 `scene` 归类
4. 多个客户端共享访问同一套数据
5. 通过 `siteId`、`sceneId` 过滤查看范围

### 2.4 不应该怎么做

当前不应该：

1. 按客户端拆库
2. 按客户端重复存一份数据
3. 把客户端当成独立数据源

### 2.5 当前结论

当前项目的数据理解应固定为：

`site -> scene -> dataAsset -> processTask / operationLog`

也就是：

1. `site` 表示现场
2. `scene` 表示现场下的场景
3. `dataAsset` 表示场景里的数据资产
4. `processTask` 表示围绕这些数据的处理任务
5. `operationLog` 表示相关操作记录

### 2.6 对后续前端联调的影响

前端后续请求应优先理解为：

1. 客户端只是查看入口
2. 查询重点是当前 `siteId`
3. 查询重点是当前 `sceneId`
4. 如果两个客户端请求条件一样，拿到的数据一样是正常现象

## 4. 2026-03-30 误区 02

### 4.1 当时的误区

容易把发布链路理解成：

1. Windows 和 Linux 的核心差别在 `kubectl / helm`
2. `kubectl / helm` 在什么系统执行，会直接决定最终部署环境

### 4.2 正确理解

当前更准确的理解应是：

1. 发布流程是：
   1. Docker 构建镜像
   2. 上传镜像到华为云 SWR
   3. Kubernetes / Helm 部署发版
2. 不同环境的核心差异，主要在“构建镜像”这一步
3. 真正关键的是：
   1. 最终镜像是谁构建的
   2. 用什么 Linux 环境构建的
4. `kubectl / helm` 更像远程控制器：
   1. 负责告诉集群使用哪个镜像
   2. 不负责决定镜像内部系统环境

### 4.3 对 Windows 和 Linux 的正确区分

1. Windows 和 Linux 在 `kubectl / helm` 层面的主要差别：
   1. shell 语法
   2. 路径格式
   3. 工具体验
   4. `grep` / `findstr` 等命令差异
2. 这些差异通常不改变部署模型本身
3. 真正更影响最终发布一致性的，是镜像构建环境

### 4.4 当前项目中的正确理解

当前项目应理解为：

1. Windows 负责：
   1. 开发
   2. 本地验证
   3. 执行 Docker / kubectl / helm 命令
2. Docker Desktop 的 Linux builder 负责：
   1. 生成最终 Linux 镜像
3. 华为云负责：
   1. 拉取镜像
   2. 运行 Linux 容器

### 4.5 当前结论

对当前项目来说，后续判断发布环境时要优先问：

1. 镜像是谁构建的
2. 镜像基于什么 Linux 环境构建

而不是先问：

1. `kubectl` 是在 Windows 还是 Linux 执行的
